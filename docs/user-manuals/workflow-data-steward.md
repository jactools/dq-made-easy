# Data Steward Workflow Guide

**Role:** Data steward responsible for governing data quality rules, resolving incidents, reviewing business definitions, and managing exceptions.
**Time to read:** 10 minutes
**Last updated:** 2026-05-31

## Responsibilities in scope

- Reviewing and approving rule changes.
- Managing rule lifecycle: activate, deprecate, supersede, or retire rules.
- Running and reviewing exception-analysis sessions.
- Reviewing and approving business term definitions.
- Triaging data quality incidents and coordinating remediation.

## Core workflows

### 1. Review and approve a rule

1. Open **Governance** → **Approvals**.
2. Find the pending rule approval in your inbox.
3. Open the rule to inspect its definition, check type, severity, scope, and metadata.
4. Approve or reject with a rationale comment.
5. The rule transitions to active automatically after approval.

### 2. Deprecate or retire a rule

1. Open **Rules** and find the rule.
2. Open the rule lifecycle actions.
3. Choose **Deprecate** with a rationale, or **Retire** if the rule is no longer valid.
4. The governance audit trail records the actor, timestamp, and reason automatically.

### 3. Run an exception-analysis session

1. Open **Rule Quality** → **Exception Analysis**.
2. Select the rule and source scope to analyse.
3. The platform generates slice batches using the configured partition strategy.
4. Submit the first slice and view the results summary.
5. Continue slices until the exception space is exhausted or the budget is reached.
6. Retrieve each slice's exception results from the stored analysis pack.

> Results are stored in object storage. The UI shows metadata and summaries only, not raw source rows.

### 4. Review a business term definition draft

1. Open **Data Catalog** → **Data Definitions**.
2. Find the draft generated by the AI-assisted definition flow or a colleague.
3. Read the canonical English definition, domain scope, disambiguations, and policy linkage.
4. Submit revision feedback if needed, or approve the draft for board review.
5. After board approval, the backend imports the definition into OpenMetadata automatically.

### 5. Triage a data quality incident

1. Open **Operations** → **Incidents**.
2. Review the open incidents for your workspace.
3. Check the incident kind, severity, scope, and correlation metadata.
4. Assign the incident to the right owner or claim it yourself.
5. Record external ticket references if the remediation is tracked in an ITSM system.
6. Mark the incident resolved when remediation is confirmed.

See the [Data Observability Triage Guide](./data-observability-triage-guide.md) for a full walkthrough.

## What to check when a rule is flagged for drift

- Has the source schema changed recently?
- Is there a null-rate shift or volume anomaly on the affected dataset?
- Is the rule's execution target still pointing at the correct dataset and version?
- Is the drift event a genuine degradation or a planned schema migration?
