# DQ Kong Gateway

Kong API Gateway service for the Data Quality Made Easy platform.

## Overview

Kong is an open-source API Gateway and Microservices Management Layer, delivering high performance and reliability. It serves as a proxy and gateway for all API traffic in the DQ platform.

## Directory Structure

```
dq-kong/
├── Dockerfile.kong           # Kong container image definition
├── scripts/
│   └── build_and_push.sh    # Build and push script
└── README.md                # This file
```

## Building

### Local Build (No Push)

```bash
./scripts/build_and_push.sh --no-push
```

### Build and Push to Registry

```bash
./scripts/build_and_push.sh
```

### Build Without Cache

```bash
./scripts/build_and_push.sh --no-cache
```

## Configuration

The build script requires these environment variables in `.env`:

- `DQ_KONG_REGISTRY` - Docker registry URL
- `DQ_KONG_NAMESPACE` - Docker namespace/organization
- `DQ_KONG_IMAGE` - Image name (e.g., "dq-kong")
- `DQ_KONG_TAG` - Image tag (auto-set by build_and_push_all.sh)

Example `.env`:

```bash
DQ_KONG_REGISTRY=docker.io/
DQ_KONG_NAMESPACE=jacbeekers/
DQ_KONG_IMAGE=dq-kong
DQ_KONG_TAG=<auto-generated>
```

## Image Details

- **Base Image**: `kong:3.9.1`
- **Ports**: 
  - 8000: HTTP Proxy
  - 8443: HTTPS Proxy  
  - 8001: Admin API
  - 8002: Kong Manager UI

## Usage

Kong is configured and seeded via:

1. **kong-migrations** container - Sets up the Kong database schema
2. **kong** (this image) - Runs the gateway
3. **bootstrap script** (`scripts/bootstrap_kong.sh`) - Configures routes/plugins, registers JWT consumers, and reconciles ACL groups from the live Keycloak Admin API

### SSO / JWT issuer notes

- Kong’s JWT plugin is configured to validate tokens using the `preferred_username` claim.
- The bootstrap script reads the live Keycloak realm through the Admin API, creates Kong consumers for current Keycloak users plus the GX worker service account, then registers a JWT credential for each consumer.
- Kong ACL groups are derived from the live realm role assignments and returned upstream in `X-Consumer-Groups`.
- The Kong container needs the Keycloak admin URL plus admin credentials in its environment so bootstrap can query the realm at startup.
- The Kong bootstrap now fails fast when required configuration is missing. Set `DQ_API_INTERNAL_URL`, `KEYCLOAK_INTERNAL_URL`, `KEYCLOAK_PUBLIC_HOSTNAME`, `KEYCLOAK_ADMIN_REALM`, `KEYCLOAK_SYSTEM_ADMIN_USERNAME`, `KEYCLOAK_SYSTEM_ADMIN_PASSWORD`, `KEYCLOAK_REALM`, `DQ_ENGINE_OIDC_CLIENT_ID`, `UI_VITE_LOCAL_URL`, `UI_NGINX_LOCAL_URL`, `TRUST_PROXY_AUTH`, `KONG_TRACING_INSTRUMENTATIONS`, `KONG_TRACING_SAMPLING_RATE`, and `KONG_OTEL_ENDPOINT` explicitly.
- `KONG_ADMIN_INTERNAL_URL` remains the only bootstrap default, because the script runs against the local Kong Admin API inside the Kong container by default.
- If you change `SSO_PUBLIC_ISSUER_URL`, `KEYCLOAK_PUBLIC_HOSTNAME`, or `scripts/bootstrap_kong.sh`, rebuild and recreate the `kong` container so the new script/env takes effect:

```bash
docker compose build kong
docker compose up -d --force-recreate kong
```

See `docker-compose.yml` for the complete configuration.

## Customization

To add custom Kong plugins or configuration:

1. Modify `Dockerfile.kong` to install plugins or add configuration
2. Rebuild: `./scripts/build_and_push.sh --no-cache`

Example:

```dockerfile
RUN luarocks install kong-plugin-custom
```
